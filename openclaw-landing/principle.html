🏗️ 工作原理

深入了解 OpenClaw 的核心架构：Gateway 与 Agent 如何协同工作

🏗️ 四层解耦架构

OpenClaw 采用经典的四层解耦架构，从外到内依次是：交互层、网关层、智能体层、执行层。这种分层设计确保了各模块职责清晰、可独立演进。

各层职责

📱 交互层

协议适配，抹平15+个消息平台的差异。核心渠道在 src/，扩展渠道在 extensions/

🌐 网关层

路由、排队、调度、鉴权、协议转换。核心源码：src/gateway/server.impl.ts

🤖 智能体层

会话管理、上下文组装、记忆系统、执行循环。源码：src/agents/

⚡ 执行层

本地节点、远程节点、Skills、沙箱隔离。源码：packages/pi-embedded/

💡 设计哲学：能用最简单的方案解决问题，就绝不引入复杂的外部工具。一个 Gateway 进程管所有。

🌐 Gateway — 核心枢纽

Gateway 是 OpenClaw 的核心进程，负责所有渠道连接、消息路由和会话管理。

Gateway 的职责

📡 渠道连接

同时连接 WhatsApp、Telegram、Discord、iMessage 等多个通讯平台

🔀 消息路由

接收消息、识别发送者、路由到对应的 Agent 会话

🧠 会话管理

维护每个用户的对话上下文，支持多会话隔离

🔌 API 接口

提供 WebSocket 和 HTTP API，支持 OpenAI 兼容格式

运行时模型

📱 通讯渠道

🌐 Gateway
→
🤖 Agent

📱 通讯渠道
  • 单一常驻进程：一个 Gateway 进程处理所有路由、控制平面和渠道连接
  • 多协议支持：WebSocket 控制/RPC、HTTP API、Control UI
  • 默认本地绑定：只绑定 loopback (127.0.0.1)，安全可靠
  • 认证保护：默认需要 token 或 password 认证

💡 默认端口是 18789，可以通过 --port 参数或配置文件修改

Lane Queue：车道式排队系统

这是 OpenClaw 最精巧的设计——每个用户会话有自己的专属车道，车道内任务严格排队。

// 核心保证：同一个用户对话，同一时刻只能有一个任务执行
// 实现方式：纯 TypeScript + Promises，无 Redis/RabbitMQ

// 每个 session key 对应一条专属车道
// 消息进来，排到自己车道的队尾
// 车道内严格先来后到
// 不同车道可以同时跑——同一条车道里，绝对一个一个来

🔄 默认串行

同一用户的任务严格排队，避免状态混乱

⚡ 显式并行

只有明确标记低风险的任务（如定时任务）才并行

Session Key 设计

session key 的结构天然带着"谁在说话"和"从哪个平台来"的信息：

agent:main:main           ← 主会话
agent:main:telegram:dm:xxx  ← Telegram 某用户私聊
agent:support:slack:ch:yyy  ← Slack 某频道的 support agent

session key 自己就是路由表 + 权限表，不需要另外搞一套复杂的权限管理系统。

🤖 Agent — AI 引擎

Agent 是 OpenClaw 连接的 AI 助手，负责理解用户意图并执行任务。

支持的 Agent

OpenClaw 目前主要支持 Pi (Inflection AI) 作为内置 Agent：

  • Pi 是一个专注于编程的 AI 助手
  • 支持工具调用、代码执行、文件操作
  • 通过 RPC 模式与 Gateway 集成
  • 支持流式响应，实时返回处理进度

Agent 的核心能力

💻 代码执行

在沙箱环境中执行代码，支持多种编程语言

📁 文件操作

读取、编辑、创建文件，智能理解项目结构

🌐 浏览器控制

自动化浏览器操作，网页抓取和测试

🔧 工具调用

调用外部 API和服务，扩展 Agent 能力

会话管理

每个用户或群组都有独立的会话：

  • 私聊会话：每个发送者独立
  • 群组会话：每个群组独立，需要 @提及 激活
  • 工作区隔离：不同项目/用途使用不同工作目录

三大死循环探测器

Agent 陷入死循环是最常见的翻车方式。OpenClaw 内置三个探测器：

🔁 重复检测

同一工具 + 同样参数反复调用 → 发现就喊停

👣 原地踏步检测

一直在查询，但结果没任何变化 → 喊停

🏓 乒乓检测

A工具→B工具→A工具→B工具，来回弹 → 喊停

详细步骤

  1. 用户发送消息 — 通过 WhatsApp、Telegram 等渠道发送
  2. 渠道协议解析 — Gateway 将渠道特定格式转换为统一消息
  3. 身份识别 — 识别发送者，确定对应会话
  4. 会话上下文 — 加载历史对话上下文
  5. Agent 调用 — 将消息发送给 Agent 处理
  6. 流式响应 — Agent 实时返回处理结果
  7. 消息发送 — Gateway 将响应发送回对应渠道

🔐 安全机制

认证体系

  • Gateway 认证：需要 token 或 password
  • 渠道白名单：可配置允许访问的号码/用户
  • 群组规则：可配置群组中是否需要 @提及

沙箱隔离

Agent 执行操作时可选启用沙箱模式：

  • 文件操作限制在指定目录
  • 代码执行在隔离容器中运行
  • 浏览器操作使用桥接 URL

三道安全防线

🔒 隔离小黑屋

代码不在本地直接跑，丢进 Docker 容器（一次性虚拟机）里跑完就扔

📋 工具白/黑名单

src/agents/tool-policy.ts 管着哪些工具能用。黑名单优先于白名单

👤 人工审批

高风险操作弹窗让你确认，点"同意"才执行

🛡️ 建议在生产环境启用沙箱模式，确保安全可控

记忆系统：Markdown 就是数据库

OpenClaw 记忆系统走了一条"反潮流"的路——Markdown 文件就是数据库。

工作区结构

~/.openclaw/workspace/
├── MEMORY.md              ← 长期记忆
├── memory/
│   ├── 2026-03-10.md     ← 今日日志（短期）
│   └── 2026-03-09.md     ← 昨日日志
├── sessions/              ← 会话存档
├── USER.md                ← 用户身份
└── SOUL.md                ← Agent 人格设定

双层记忆结构

  • 短期记忆：memory/YYYY-MM-DD.md，每天流水日志，只追加不修改
  • 长期记忆：MEMORY.md，整理过的重要记忆

每次开始新对话，Agent 自动加载今天和昨天的日志。为什么是两天？保持上下文连续性，又不撑爆窗口。

混合检索策略

两条路同时搜，7:3 比例混合：

  • 语义搜索（向量权重 0.7）：理解话里的"意思"，即使原文没关键词也能找到
  • 关键词搜索（BM25权重 0.3）：精确匹配字词

记忆瘦身：瘦身前先备忘

上下文窗口快满时，OpenClaw 会做"瘦身"。但它不会直接压缩——而是先让 Agent 做一次"紧急备忘"：

{
  "compaction": {
    "memoryFlush": {
      "enabled": true,
      "softThresholdTokens": 4000,
      "prompt": "Write any lasting notes to memory/YYYY-MM-DD.md"
    }
  }
}

这就是很多人抱怨"它忘了我说过的话"的根因——这个功能没打开。

💡 记忆透明可审计——随时打开文件，看清它记住了什么、怎么执行的

⚡ 热重载

Gateway 支持配置热重载，无需重启服务即可应用配置变更：

# 热重载模式配置
gateway.reload.mode: hybrid  # 默认选项

# 可选值：
- off      : 不重载配置
- hot      : 仅应用热安全变更
- restart  : 需要重启的变化
- hybrid   : 自动选择最佳方式

使用 openclaw gateway restart 或 openclaw secrets reload 手动重载。

🧩 插件化重构

2026 年初，OpenClaw 通过 PR #661 完成了重大插件化重构，这是架构演进的分水岭。

重构前的问题

添加一个新模型提供商需要修改 4 个核心文件：继承 BaseProvider、在 providers/index.ts 手动注册、在 model-router.ts 添加路由分支、更新配置 Schema。

重构后：接口标准化 + 动态加载

// 路由从 O(n) 降为 O(1)
async route(model: string, ...args) {
  const [providerName] = model.split('/');
  const provider = this.loader.getProvider(providerName);
  return provider.chat(...args);
}
